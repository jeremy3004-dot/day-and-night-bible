<research_summary>
## Summary

The codebase already has most primitives needed for this phase. Translation capability is modeled centrally in [`src/constants/translations.ts`](/Users/dev/Projects/EveryBible/src/constants/translations.ts), chapter-level audio download and offline lookup already exist in [`src/services/audio/audioDownloadService.ts`](/Users/dev/Projects/EveryBible/src/services/audio/audioDownloadService.ts), and no-text + audio-first rendering behavior already exists in [`src/services/bible/presentation.ts`](/Users/dev/Projects/EveryBible/src/services/bible/presentation.ts) and [`src/screens/bible/BibleReaderScreen.tsx`](/Users/dev/Projects/EveryBible/src/screens/bible/BibleReaderScreen.tsx).

The highest-value phase approach is targeted cohesion work, not a rewrite:
1. Add a dedicated audio-only translation profile (`hasText=false`, `hasAudio=true`) for manual QA.
2. Tighten selection logic so audio-only translations are usable for testing (not blocked by text-download assumptions).
3. Keep UI behavior explicit: when text is absent but audio is available, use audio-first states and avoid generic "no verses" fallbacks.

This work maps directly to READ-04, AUDIO-03, and the new AUDIO-04 requirement.
</research_summary>

<architecture_patterns>
## Architecture Patterns

### Pattern 1: Capability-first translation decisions
Use a small pure model to decide whether translation selection is allowed based on text/audio capability and offline/remote availability.

### Pattern 2: Reuse existing audio download pipeline
Do not create a new downloader; keep all chapter/translation downloads flowing through existing audio services.

### Pattern 3: No-text graceful rendering
Continue relying on `getChapterPresentationMode` and related presentation helpers so no-text chapters route into audio-first UI when audio is playable.

### Anti-patterns to avoid
- Tying translation selection strictly to `isDownloaded` when audio-only testing is needed.
- Mixing business rules directly into large screen components without model tests.
- Introducing backend dependencies for what is a local capability/presentation phase.
</architecture_patterns>

<validation_architecture>
## Validation Architecture

Automated coverage should prove capability and presentation rules without relying only on device QA:
- translation selection/eligibility model tests for audio-only scenarios
- presentation tests for no-text + audio-available mode
- audio availability/download regression tests for offline replay readiness

Manual checks should validate real runtime behavior:
- select the audio-only translation
- download audio for at least one book/translation
- open reader with no text and confirm audio-first behavior
- verify offline playback after network disable
</validation_architecture>

<open_questions>
## Open Questions

1. Should audio-only translation selection be enabled before any audio is downloaded, or only once remote audio availability is confirmed?
2. Should the translation chip explicitly show an "Audio only" badge in reader and browser surfaces?

Recommended default: allow selection when audio is remotely available or offline audio already exists, and show a concise "Audio only" cue.
</open_questions>
