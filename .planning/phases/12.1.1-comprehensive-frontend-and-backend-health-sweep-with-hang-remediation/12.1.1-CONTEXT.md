# Phase 12.1.1: Comprehensive frontend and backend health sweep with hang remediation - Context

**Gathered:** 2026-03-21
**Status:** In progress
**Source:** User request for a comprehensive health check after the premium reader/TestFlight release

<domain>
## Phase Boundary

Run an evidence-first health sweep across the shipped mobile app and its Supabase backend, then remediate concrete breakages, hangs, or high-confidence reliability problems discovered during that sweep.

Scope includes:

- running the existing automated frontend verification suite to establish baseline health
- smoke-testing the native iOS app to catch runtime hangs or broken navigation paths that source tests miss
- inspecting Supabase project health, advisors, and recent API logs for backend correctness and performance issues
- fixing the highest-confidence issues discovered during the sweep with regression coverage first

Out of scope:

- broad product redesign work unrelated to health or reliability
- speculative refactors without reproduced bugs or concrete advisor evidence
- new feature work beyond what is required to restore dependable behavior
</domain>

<decisions>
## Implementation Decisions

### Locked Decisions

- This sweep runs in a clean git worktree so unrelated local changes in the main workspace stay untouched.
- Runtime/frontend issues must be reproduced before implementation and should gain automated regression coverage first.
- Backend fixes should be evidence-led: prioritize advisory findings or reproducible API/runtime issues over speculative database changes.
- Prefer small, production-safe fixes over broad architecture changes during this sweep.

### Claude's Discretion

- Which smoke-test path best samples the current release-critical experience as long as it covers startup, Bible entry, chapter open, and reader/listener continuity.
- Whether backend remediation lands as one migration or a small set of migrations, as long as the changes stay minimal and verifiable.
</decisions>

<specifics>
## Specific Findings So Far

- Automated baseline already passes in this worktree: `npm run release:verify` and `npm test`.
- Supabase project `EveryBible` (`ganmududzdzpruvdulkg`) is `ACTIVE_HEALTHY` and recent API logs do not show an obvious live outage.
- Supabase advisors still report actionable hygiene issues:
  - RLS policies repeatedly call `auth.*` functions in ways that trigger `auth_rls_initplan` warnings.
  - `public.group_sessions.created_by` lacks a covering index for its foreign key.
  - Auth security settings warn about leaked-password protection and MFA breadth, which may require config/dashboard action rather than app-code changes.
- The biggest remaining blind spot is native runtime/device behavior after the recent reader and Bible data changes.
</specifics>

<deferred>
## Deferred Ideas

- Broader backend config policy hardening beyond the findings surfaced by this sweep
- Full end-to-end automation of native Bible journeys if current smoke checks reveal enough stable selectors to justify it
</deferred>
