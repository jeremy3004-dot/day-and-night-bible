<task type="auto">
  <name>Task 1: Fix tab bar height to include home-indicator clearance</name>
  <files>src/design/system.ts, src/navigation/TabNavigator.tsx</files>
  <action>
The tab bar height of 72 plus `paddingBottom: spacing.sm` (8) is not enough for iPhones with a
home indicator (~34 pt safe-area bottom inset). The tab bar height must absorb the bottom inset so
labels are never clipped.

In `src/navigation/TabNavigator.tsx`:
- Import `useSafeAreaInsets` from `react-native-safe-area-context`
- Inside `TabNavigator`, call `const insets = useSafeAreaInsets()`
- Update `defaultTabBarStyle` to use a dynamic height that accounts for the bottom inset:

```ts
const tabBarHeight = layout.tabBarBaseHeight + insets.bottom;
const defaultTabBarStyle = {
  backgroundColor: colors.cardBackground,
  borderTopColor: colors.cardBorder,
  borderTopWidth: 1,
  paddingTop: spacing.sm,
  paddingBottom: insets.bottom > 0 ? insets.bottom : spacing.sm,
  height: tabBarHeight,
} as const;
```

In `src/design/system.ts`:
- Rename the existing `tabBarHeight: 72` field to `tabBarBaseHeight: 56` — this is the icon + label
  area without the home-indicator clearance. 56 pt gives comfortable icon/label spacing before the
  inset is added.
- Keep the `layout` export shape so callers that use `layout.tabBarBaseHeight` compile cleanly.

IMPORTANT: Check every existing reference to `layout.tabBarHeight` in the codebase:
- `src/screens/learn/ReadingPlanDetailScreen.tsx` (line 659): `paddingBottom: layout.tabBarHeight + spacing.xl`
- `src/screens/learn/ReadingPlanListScreen.tsx` (line 631): `paddingBottom: layout.tabBarHeight + spacing.xl`
- `src/screens/more/TranslationBrowserScreen.tsx` (line 457): `paddingBottom: layout.tabBarHeight + spacing.xl`
- `src/components/audio/MiniPlayer.tsx` (line 130): `bottom: 74` — fixed below in Task 3

Update all of these to use `layout.tabBarBaseHeight` so they no longer use the old constant.
The dynamic height in TabNavigator is computed at runtime; consumers that hard-code scroll
offsets should use `tabBarBaseHeight` as a safe minimum.

### Fix A — PIN keyboard type (PrivacyPreferencesScreen.tsx lines 150–174)
Both `TextInput` elements for `pinInput` and `pinConfirmation` show the full keyboard. Change both:
- Add `keyboardType="number-pad"` — shows numeric keypad only, no letters
- Add `secureTextEntry` — masks the digits as they are typed
- Remove `autoCapitalize="none"` and `autoCorrect={false}` — not relevant for number-pad inputs

```tsx
<TextInput
  value={pinInput}
  onChangeText={(value) => { setPinInput(value); setErrorKey(null); }}
  placeholder={t('onboarding.pinPlaceholder')}
  placeholderTextColor={colors.secondaryText}
  style={styles.input}
  keyboardType="number-pad"
  secureTextEntry
/>

<TextInput
  value={pinConfirmation}
  onChangeText={(value) => { setPinConfirmation(value); setErrorKey(null); }}
  placeholder={t('onboarding.pinConfirmPlaceholder')}
  placeholderTextColor={colors.secondaryText}
  style={styles.input}
  keyboardType="number-pad"
  secureTextEntry
/>
```

### Fix B — Legacy Bible reader header top padding (BibleReaderScreen.tsx)
The `header` style has `paddingTop: 6` (line ~2088). The screen is wrapped in
`<SafeAreaView edges={['top', 'bottom']}>`, which means the status bar inset is provided by
SafeAreaView — but 6 pt is still too little visual breathing room between the safe-area edge and
the header row. Increase `paddingTop` in the `header` style to `spacing.md` (12):

```ts
header: {
  flexDirection: 'row',
  alignItems: 'center',
  justifyContent: 'space-between',
  paddingHorizontal: 16,
  paddingTop: spacing.md,   // was 6
  paddingBottom: 14,
  borderBottomWidth: 1,
},
```

There is also a `minimalHeader` override with `paddingBottom: 8` — leave that unchanged.

Also check the second occurrence of `paddingTop: 6` near line 2482 in the same file (the
`followAlongHeader` or a similar secondary header) and apply the same fix there if it exists.

### Fix C — AnnotationsScreen safe area (AnnotationsScreen.tsx)
The screen renders into a plain `<View>` with no `SafeAreaView`, so its header sits under the
status bar on most iPhones. Fix:

1. Add the import:
   ```tsx
   import { useSafeAreaInsets } from 'react-native-safe-area-context';
   ```

2. Inside `AnnotationsScreen`, call:
   ```tsx
   const insets = useSafeAreaInsets();
   ```

3. In the `container` style application, add top padding dynamically:
   ```tsx
   <View style={[styles.container, { backgroundColor: colors.background, paddingTop: insets.top }]}>
   ```

4. In the `header` style, remove the hardcoded `paddingTop: spacing.lg` (16) so we don't
   double-count — the inset is now on the outer container:
   ```ts
   header: {
     flexDirection: 'row',
     alignItems: 'center',
     justifyContent: 'space-between',
     paddingHorizontal: layout.screenPadding,
     paddingTop: spacing.sm,    // was spacing.lg (16)
     paddingBottom: spacing.md,
   },
   ```

<task type="auto">
  <name>Task 3: Fix MiniPlayer bottom position to clear the tab bar</name>
  <files>src/components/audio/MiniPlayer.tsx</files>
  <action>
The mini player shell uses `bottom: 74` (hardcoded). On phones where the tab bar is taller
(because `insets.bottom` pushes the height above 72), the player can overlap tab bar icons.

The shell must sit above the actual rendered tab bar. Compute the correct bottom offset at runtime:

1. Import `useSafeAreaInsets` from `react-native-safe-area-context`.

2. Inside `MiniPlayer`, call:
   ```tsx
   const insets = useSafeAreaInsets();
   ```

3. Calculate the live tab bar height (matching what TabNavigator now computes):
   ```tsx
   const tabBarHeight = layout.tabBarBaseHeight + insets.bottom;
   ```

4. Apply it to the shell style as an inline prop instead of the static StyleSheet value:
   ```tsx
   <View pointerEvents="box-none" style={[styles.shell, { bottom: tabBarHeight + spacing.sm }]}>
   ```
   `spacing.sm` (8) adds a small gap between the player card and the tab bar top edge.

5. Remove the hardcoded `bottom: 74` from `styles.shell` in the `StyleSheet.create` block.

The `container` style inside the shell is unchanged — only the outer `shell` bottom is affected.
  </action>
  <verify>npm run typecheck</verify>
  <done>
- TypeScript compiles with no errors
- `styles.shell` no longer has a hardcoded `bottom`
- Shell bottom is computed as `tabBarBaseHeight + insets.bottom + spacing.sm`
  </done>
